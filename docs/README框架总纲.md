# README 框架总纲

这份文档不是在教人立刻写某一份 README，而是在总结一套以后可以反复复用的框架。

适用对象：

- 项目门面生成器这类 skill
- 单项目仓库 README
- 多项目仓库 README
- 技能集合仓库 README
- 需要“先调查，再产出，再过审，再收尾”的文档型 skill

## 一句话主链

以后做这类 README，先记住这条主链：

`调查 -> 定位 -> 生成 -> 评测 -> 放行 -> 收尾`

它不是“直接写文档”，而是“先建立判断，再产出结果，再决定是否继续自动化”。

这里的前两步不要混：

- 调查：收集当前项目、仓库和上下文的客观事实
- 定位：基于这些事实，确定本次 README 的层级、类型、目标读者、输出目标与执行模式

最短理解：

- 调查 = 看清楚现在是什么情况
- 定位 = 想明白接下来该怎么处理

## 道

README 的第一性原理不是把作者脑子里的所有想法都倒出来，而是优先回答一个陌生读者最先关心的问题。

对人看的 README，前半部分优先解决这些判断：

- 这是什么
- 对我有没有用
- 我现在能不能用
- 用完能到什么程度

对这类 skill 来说，还要再加一层：

- 人负责质量与放行
- AI 负责调查、整理、补件、执行
- Git 负责文件安全与可回退

所以这套框架的核心不是“让 AI 代替人”，而是：

`人做决策，AI 做执行，Git 保安全。`

## 法

### 1. 先判断仓库形态

不要默认“一仓库一项目”。

优先判断：

- 单项目仓库
- 多项目仓库
- 技能集合仓库
- 文档 / 知识库仓库
- 模板 / 资源仓库

### 2. 再判断生成层级

同一个仓库里，README 也可能有不同层级。

优先判断：

- 仓库根 README
- 子项目 README
- 单个 skill README

### 3. 再判断项目类型

不同项目类型，README 前半部分排序不同。

常见类型：

- Skill / 工具 / 小产品
- 知识 / 理论 / 教程
- 工程 / 库 / 开源项目

### 4. 再决定执行模式

推荐统一成三种模式：

- `审核后自动`
- `自动流`
- `手动流`

默认优先用 `审核后自动`。

## 术

### 1. 上下文策略

不要要求用户先把上下文讲满。

优先顺序：

1. 先使用当前已有上下文
2. 若不足，补最小必要上下文
3. 只读关键文件，不无差别扫全仓库
4. 只有判断会明显影响结果时，才最小化向用户确认

### 2. README 前半部分排序策略

#### Skill / 工具 / 小产品

推荐顺序：

1. 有什么用
2. 对谁有用
3. 适用范围
4. 怎么用
5. 使用效果

#### 知识 / 理论 / 教程

推荐顺序：

1. 解决什么认知问题
2. 核心观点
3. 对谁有用
4. 能得到什么
5. 怎么读或怎么实践

#### 工程 / 库 / 开源项目

推荐顺序：

1. 这是什么
2. 能做什么
3. 对谁有用
4. 安装 / 快速开始
5. 示例 / 核心能力

### 3. README 评测策略

README 不是写完就结束，而是写完先过审。

固定走这一步：

`README 生成 -> README 评测 -> 放行`

评测重点：

- 是否先回答读者最关心的问题
- 是否写对仓库层级与项目类型
- 安装、使用、示例是否可执行
- 是否有明显遗漏或冲突
- 是否有过度承诺或脱离项目实际的内容

评测结果统一成三种：

- `绿色`：通过，继续后续流程
- `黄色`：可用，但先自动修一轮再复评
- `红色`：方向不对或信息不足，停下来人工确认

### 4. 收尾策略

推荐固定成：

`README -> README 评测 -> LICENSE -> Git`

规则：

- README 评测没通过，不进入 LICENSE
- LICENSE 没明确，不进入自动推送
- Git 只提交本次门面整理相关文件
- 提交边界不清，先停下来确认

## 器

做这套框架时，至少准备这几样“器”：

- `skill.md`：给 AI 的执行规则
- `README.md`：给人看的产品页
- 安装模板：命令安装 + 让 AI 帮你安装
- 项目类型模板：不同类型的最小结构
- README 评测卡：绿色 / 黄色 / 红色
- Git 护栏：自动提交边界与停机条件

## 从上到下怎么造一个 skill

如果以后你要继续造类似 skill，可以按这个顺序组织：

1. `name / description / trigger`
2. 一句正式定义
3. 执行原则
4. 执行流程
5. 判断原则
6. 入口与执行模式
7. 最小必要上下文
8. 主产物规则
9. 项目类型模板
10. 评测与放行
11. 收尾规则
12. 输出

其中最关键的是三段：

- 判断：先判断仓库形态、层级、项目类型
- 评测：不是生成完就算完成
- 收尾：只有通过后才继续自动化

## 这套框架为什么适合继续延伸

它不只适合 README。

只要一个 skill 的目标是：

- 先调查
- 再产出
- 再过审
- 再决定是否继续自动化

那都可以套这套骨架。

例如可以延伸成：

- 文档门面生成器
- 仓库首页生成器
- 阶段总结生成器
- 发布材料生成器
- 知识主页生成器

## 当前最值得继续深化的点

这套框架已经能用，但还有几个值得继续收紧的地方：

- 把“推荐模式”做成固定建议卡格式
- 把“README 评测”做成固定评测卡格式
- 给不同项目类型补更硬的最小模板
- 把自动提交边界写成更机械的文件范围规则
- 继续明确“项目门面”到底止于哪里，不膨胀成发布管理器

## 最后的压缩版

如果以后只想记最短的一版，就记这几句：

- README 前半部分，优先回答目标读者最先关心的问题
- 先判断仓库形态、生成层级、项目类型，再写
- 先调查，再推荐模式
- 先生成，再评测，再放行
- 人保质量，AI 提效率，Git 保安全
