把“已经做出来的作品”，整理成适合发到 X 的主帖、跟帖和配图文案。  
它不是泛用发帖器，而是一个更接近“秋知”表达方式的创作者发帖 skill：真实、简洁、不过度营销，但会先判断这条帖更该先讲结果、动机，还是前后变化。

## 它有什么用

很多作品不是做不出来，而是做出来之后，没有一个像样的门面和发帖入口，就很难被真正看见。  
这个 skill 解决的不是“写一段宣传文案”，而是把一条发布帖整理到更适合传播的状态。

它优先帮你补这些东西：

- 主帖
- 第一条带链接跟帖
- 补充解释跟帖
- 配图版短文案
- 海报图标题文案

## 对谁有用

它更适合这些人：

- 做了项目、skill、工具或作品，准备发 X 的创作者
- 不想每次都从零打磨主帖和跟帖的人
- 已经有截图、README、GitHub 链接，但还不知道怎么发的人
- 想保持本人视角，不想把文案写成官号说明书的人

## 适用范围

它最适合这些场景：

- 开源一个新项目
- 发布一个新 skill
- 分享一个刚做完的作品
- 配 README 截图发帖
- 配结果图、成品图、对比图发帖
- 先发主帖，再用跟帖补链接和解释

它默认输出到这个程度：

- 最低：主帖 + 一条带链接跟帖
- 常规：主帖 + 2 到 3 条跟帖
- 可选：更短配图版 / 海报文案版

## 它是怎么工作的

它不是直接开写，而是先判断这次帖子的对象和发法。

默认顺序是：

1. 先调查项目、截图、链接和发帖目标
2. 先判断更适合写结果展示帖、动机帖，还是对比帖
3. 再判断这次更适合主帖、线程，还是配图版
4. 先生成主帖
5. 再生成带链接的跟帖
6. 最后再检查主帖有没有太满、太像说明书、太像营销稿

## 它会先判断写哪种帖

这个 skill 现在不会默认把所有帖都写成“我为什么做它”。

它会优先判断三种重心：

- `结果展示帖`
  适合有明确产物、支持范围、结果图、成品图时
- `动机帖`
  适合作者视角更强、想先讲为什么做时
- `对比帖`
  适合有 before / after、输入输出差异、整理前后对照时

默认规则是：

- 冷启动发布且有明确结果时，优先 `结果展示帖`
- 有强烈创作者动机时，可用 `动机帖`
- 有明显变化时，优先 `对比帖`

也就是说，秋知风格不是“永远先讲动机”，而是“保持创作者口吻，但先讲最该先讲的那层信息”。

## 秋知风格

这不是泛用品牌文案器。  
默认采用的是更接近“秋知”的创作者表达：

- 先讲为什么做
- 再讲做了什么
- 说人话
- 不装腔
- 不拔高
- 不把主帖写成 README 摘抄

## 推荐发法

默认更推荐这两种：

- 结果展示主帖：先把它做成了什么、支持什么、能帮谁说清楚
- 配图主帖：主帖更短，截图承担“这是什么”
- 主帖 + 跟帖：主帖负责钩子，跟帖负责链接和补充

通常不建议把链接塞进已经很满的主帖里。  
更稳的方式是：主帖配图，第一条跟帖放链接。

## 输出示例

默认优先给：

- 主帖
- 第一条带链接跟帖
- 如有需要，再补 1 到 2 条解释跟帖

可选再给：

- 更短的配图版主帖
- 更像海报标题的图文案

例如，针对“项目门面生成器”这种开源发布，主帖可以写成：

```text
项目门面生成器

把一个已经做出来的项目，
整理到能见人、能介绍、能发出去的状态。

会补：
README
安装方式
对外介绍
以及必要的 LICENSE / Git 收尾
```

第一条带链接跟帖可以写成：

```text
GitHub：
https://github.com/AdgaiWalker/creator-publish-workflow/tree/main/%E9%A1%B9%E7%9B%AE%E9%97%A8%E9%9D%A2%E7%94%9F%E6%88%90%E5%99%A8
```

## 设计原则

```text
先调查再写
先判断该先讲结果还是先讲动机
冷启动时优先让人先看到结果
主帖短，跟帖补
图片承担“这是什么”
文案承担“为什么值得看”
人味优先，营销腔后退
```
