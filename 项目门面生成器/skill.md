# 项目门面生成器

在项目完成后或阶段收尾时，先基于已有或补齐后的最小必要上下文完成调查并推荐执行模式，再以 README 为核心整理出可展示、可介绍、可移交的项目门面材料；并按确认后的策略处理安装方式、遗漏检查、LICENSE 与 Git 收尾，但不越权代替用户做最终发布决策。

## 执行原则

- 人只负责决策，AI 负责分析、生成、整理与执行
- AI 负责整理、补件、送审；人负责审核、放行或打回
- 不问 AI 自己能判断的问题
- 启用后先调查当前状态，再推荐执行模式，不默认立刻改文件
- 优先使用当前对话与工作区中的已有上下文
- 若上下文不足，先补齐最小必要上下文，再继续生成
- 仅在关键信息缺失时，用少量选项提问
- README 完成后，不立刻结束，要继续自动推进后续步骤
- README 生成后先评测，评测通过后再继续 LICENSE 与 Git
- README 档位、LICENSE 策略、推送策略一旦明确，后续步骤自动串行执行

## 适用前提

启用后，若当前已存在足够上下文，可直接进入调查与模式推荐阶段，例如：

- 已读取项目代码、文档或已有 README
- 已理解项目用途、受众或用户目标

若上下文不足，则先在当前工作区补齐最小必要上下文，再继续调查与执行。

## 执行流程

1. 先调查当前仓库状态并推荐执行模式
2. 判断当前上下文是否充分
3. 若不足，先读取工作区关键文件补齐最小必要上下文
4. 自动判断仓库形态与本次生成层级
5. 基于上下文判断项目类型
6. 生成或优化 README 作为主产物
7. 检查遗漏项并评测 README
8. README 评测通过后，自动检查并处理 LICENSE
9. 在 README 与 LICENSE 均通过后，自动执行 Git 收尾并推送到远程仓库

## 判断原则

- 仓库形态、生成层级、项目类型等判断，优先由 AI 自动完成，不默认交给用户
- 不默认“一仓库一项目”，避免把集合仓库误写成单体项目
- 仓库形态可优先判断为：单项目仓库、多项目仓库、技能集合仓库、文档 / 知识库仓库、模板 / 资源仓库
- 生成层级可优先判断为：仓库根 README、子项目 README、单个 skill README
- 只有当判断结果会明显影响输出方向，且无法从当前上下文稳定推断时，才最小化向用户确认

## 入口与执行模式

启用本 skill 后，先做一次快速调查，不直接进入写入。

优先调查：

- 当前上下文是否充分
- 当前仓库形态与本次生成层级
- README 状态
- LICENSE 状态
- Git 与远程仓库状态

根据调查结果，优先推荐以下三种模式之一：

- 审核后自动：先给出建议卡，用户确认一次后自动跑完后续步骤
- 自动流：当 README 档位、LICENSE 策略、推送策略都已明确时，直接自动执行
- 手动流：每个关键阶段都停下来让用户确认

默认优先推荐“审核后自动”。

推荐表达应尽量简洁，例如：

- 推荐模式：审核后自动
- README：需要重做或优化
- LICENSE：沿用现有 / 建议补充某种许可证
- Git：远程明确，完成后可自动推送

用户只需做一次模式决策，例如：

- 按推荐执行
- 改成自动流
- 改成手动流

### 最小必要上下文

当上下文不足时，优先读取能快速建立项目认知的关键文件，例如：

- 顶层 `README.md`
- 依赖或项目定义文件，如 `package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`
- 主要源码目录
- 文档目录
- 示例目录
- 配置文件

只读取建立最小必要上下文所需的关键文件，避免无差别扫描整个项目。

## README 规则

按项目类型选择最小必要结构，不堆无关章节。

在判断项目类型之前，先判断当前任务目标是：

- 仓库根 README
- 子项目 README
- 单个 skill README

若为技能集合仓库或多项目仓库，根 README 应写成总入口；单个目录下的 README 再按对应项目类型处理。

常见类型：

- 库 / 框架
- CLI 工具
- 网站 / 应用
- Skills
- 理论框架
- 文档 / 教程

常见内容：

- 标题 + 一句话介绍
- 安装或使用方法
- 核心亮点或示例
- API / FAQ / 应用场景，仅在需要时加入
- License 说明，仅在已明确或后续需要补充时加入
- 若目标是 Skills，默认补上安装方式与最短调用示例

### 前半部分排序规则

README 前半部分不要写死成一种顺序，而应优先回答目标读者最先关心的问题。

先判断：

- 当前项目属于 Skill / 工具 / 小产品，还是知识 / 理论 / 教程，还是工程 / 库 / 开源项目
- 目标读者进入 README 时，最先关心的是“能不能用”“值不值得读”还是“能不能快速跑起来”

然后再决定前半部分的排序。

推荐顺序：

- Skill / 工具 / 小产品：有什么用 -> 对谁有用 -> 适用范围 -> 怎么用 -> 使用效果
- 知识 / 理论 / 教程：解决什么认知问题 -> 核心观点 -> 对谁有用 -> 能得到什么 -> 怎么读或怎么实践
- 工程 / 库 / 开源项目：这是什么 -> 能做什么 -> 对谁有用 -> 安装 / 快速开始 -> 示例 / 核心能力

规则：

- 先让用户判断“这是不是给我用的”，再展开机制和原理
- “设计原则”“第一性原理”“为什么这样设计”通常放在前面判断完成之后，不抢第一屏
- 如果用户目标是先让更多人用，再让人理解设计思想，则原则部分应后置，但不要完全删掉

如项目偏正式发布、品牌展示或社区传播，可优先使用“封面式开头”。

### 封面式开头模板

适用：正式发布、开源项目首页、需要更强展示感的 README。

推荐结构：

    横幅图或品牌图
    H1 标题
    按钮式徽章行
    1 到 2 段高密度介绍

开头规则：

- 优先使用已有横幅图、Logo、截图或品牌素材
- 若项目无现成图片，不虚构外链图片，可退化为标题 + 徽章 + 简介
- 徽章优先放核心跳转，如 Docs、官网、License、社区、发布方
- 开头文案要短，先讲项目是什么，再讲它为什么值得用
- 若采用封面式开头，正文第一屏不要再堆大段背景介绍

## 案例

### 案例 1：技能集合仓库

以 `Skills-Walker` 为例：

- 仓库形态：技能集合仓库
- 仓库根 README：写成集合入口页，重点放仓库定位、快速使用、推荐入口、技能索引、文档入口
- `创作/项目门面生成器/README.md`：写成单个 skill 的介绍页，重点放解决什么问题、安装方式、最短调用、工作方式、遗漏检查
- `创作/项目门面生成器/skill.md`：写成执行规则页，重点放上下文补齐、判断逻辑、README 生成、LICENSE / Git 收尾建议

### 案例 2：单项目仓库

- 仓库形态：一个仓库主要服务一个项目
- 处理策略：根 README 直接作为项目门面，不再额外拆集合入口页
- 输出重点：项目介绍、安装、使用、示例、License

### 案例 3：多项目仓库

- 仓库形态：一个仓库包含多个独立子项目或模块
- 处理策略：根 README 写成总入口与项目地图，每个子项目再分别生成自己的 README
- 规则重点：不把整个仓库误写成一个单体项目

## 提问规则

仅在缺少关键决策时，用少量选项提问。

可补问：

- 目标用户：开发者 / 决策者 / 普通用户 / AI 用户 / 思考者
- 风格：极简 / 标准

如上下文已足够，优先先完成调查并推荐模式，再在一次确认后自动推进，不再零散追问。

## 填充原则

- 如无必要勿增实体
- 保留最小可行示例
- 语言简洁，直击痛点
- 优先保留已有信息，再做重构与美化
- 若许可证尚未明确，不擅自写死具体许可证类型
- 若目标是 Skills，安装方式优先从上下文直接推断，不把本可推断的信息继续丢给用户决策

## Skills 安装规则

当目标是 Skills 类型时，README 默认包含 `安装` 部分，不要省略。

优先输出两种安装方式：

1. 命令安装
2. 让 AI 帮你安装

### 方式 1：命令安装

若上下文中已能判断仓库地址与 skill 名称，直接生成可复制命令：

```bash
npx skills add <owner/repo> --skill <skill-name>
```

### 方式 2：让 AI 帮你安装

提供一段可以直接贴给 AI 的提示词，默认包含：

- 仓库链接
- skill 名称
- 优先使用命令安装
- 安装完成后告知最短调用方式

参考结构：

```text
帮我安装这个 skill：
<repo-url>

skill 名称：<skill-name>

请帮我用合适的方式完成安装。
如果支持命令安装，优先使用：
npx skills add <owner/repo> --skill <skill-name>

安装完成后，告诉我最短怎么调用它。
```

规则：

- 仓库地址、skill 名称、安装命令能从上下文推断时，直接补全，不再追问
- 只有仓库地址或 skill 名称无法判断时，才最小化补问
- 给人看的 README 优先保证“能复制、能看懂、能直接交给 AI”

## 美化规则

已有 README 时：

1. 保留核心信息
2. 调整结构和层级
3. 按所选风格统一视觉层次
4. 补齐必要但缺失的内容

风格：

- 极简：少徽章、少装饰、正文优先
- 标准：适量徽章、清晰分节、表格与引用适度使用
- 标准风格在合适时可使用“封面式开头”

## 自动工作流

README 完成后，不要直接结束。

在调查和模式确认完成后，优先按预设策略自动串行执行，而不是每一步都停下来询问用户。

用户只需在开始时，或首次使用时，尽量一次性明确这些决策项：

- README 档位：快速 / 标准 / 发布
- LICENSE 策略：沿用现有 / 固定许可证 / 由 AI 推荐并直接应用 / 始终询问
- 推送策略：不推送 / 只提交本地 / 远程明确时自动推送

若用户没有逐项说明，但表达了“正式发布”“整理门面”“推到远程”等明显意图，可优先推荐“审核后自动”，并在用户一次确认后按发布模式执行。

默认顺序：

1. 生成或优化 README
2. 先进行 README 评测
3. 若评测为绿色，则继续检查 LICENSE 状态
4. 若评测为黄色，则先自动修正一轮再复评
5. 若 README 与 LICENSE 均已就绪且通过，则自动执行 Git 收尾

AI 先自行检查：

- 项目名称与一句话介绍是否完整
- 安装方式是否缺失
- 使用方法是否缺失
- 示例、截图、演示入口是否缺失
- 是否已有 LICENSE
- README 中是否已有 License 说明
- 当前目录是否为 Git 仓库
- 是否已配置远程仓库
- 是否存在可提交改动
- 当前分支是否已有明确的远程推送目标
- 当前工作区是否存在与本次门面整理无关的改动

## README 评测与放行

README 生成完成后，不直接进入 LICENSE 或 Git。

先进行一次 README 评测，确认其是否达到可展示、可介绍、可移交的最低标准。

评测重点：

- 是否先回答目标读者最关心的问题
- 是否写对当前仓库层级与项目类型
- 安装、使用、示例是否可执行
- 是否存在明显遗漏或冲突
- 是否有过度承诺或脱离项目实际的内容

评测结果分为三种：

- 绿色：通过，继续后续流程
- 黄色：可用但需要修正，先自动修正后再复评
- 红色：不通过，停止自动推进并提示用户确认

只有 README 评测通过后，才继续处理 LICENSE 与 Git。

## LICENSE 规则

若需处理 LICENSE：

- 优先基于上下文推荐，而不是要求用户自行判断
- 常见选项仅保留：
- MIT：宽松，适合大多数项目
- Apache-2.0：宽松，含专利授权
- GPL-3.0：衍生发布需继续开源
- 暂不添加 LICENSE：保留版权，不主动授权

- 若仓库中已经存在 LICENSE，则优先沿用现有许可证，同步 README 中的 License 说明后继续，不额外追问
- 若仓库中不存在 LICENSE，但当前策略已明确，例如固定 MIT 或“由 AI 推荐并直接应用”，则直接处理并继续
- 只有当许可证缺失，且当前授权策略会明显影响结果时，才最小化向用户确认一次

处理完成后：

- 生成或更新 LICENSE 文件
- 同步更新 README 中的 License 说明

## Git 规则

若需处理 Git：

- 检查当前目录是否为 Git 仓库
- 检查是否存在远程仓库或当前分支的明确上游
- 检查是否存在可提交改动
- 检查本次门面整理改动是否能与其他改动清晰区分
- README 评测通过且 LICENSE 已就绪时，优先自动执行提交
- 仅自动提交本次门面整理相关文件
- 若远程目标明确，则自动推送到远程仓库，不限定 GitHub
- 若工作区存在无关改动、混合改动或提交边界不清，则先停下来确认，不自动提交或推送
- 只有在远程目标不明确、存在多个 remote、缺少上游分支，或 push 失败时，才最小化向用户确认
- 若策略仅允许本地提交，则执行 commit，不执行 push

## 输出

- 完整流程：README + README 评测 + LICENSE 处理 + Git 收尾，按策略自动执行
- 美化 README：只优化 README
- 优化 README：重构内容并美化
- 格式：Markdown，可直接使用
