把项目整理到“能见人、能介绍、能移交”的状态。  
它先调查当前仓库状态并推荐执行模式，再以 `README` 为核心产物整理项目门面，并按策略处理安装方式、遗漏检查、`LICENSE / Git` 收尾。

## 它有什么用

很多 README 不是不会写，而是写的时候上下文不完整、结构不稳定、遗漏太多。  
这个 skill 的作用不是单纯“写一份 README”，而是在你已经做完项目，或者刚做完一个模块时，把项目门面收整成可展示、可介绍、可移交的状态。

它会优先帮你完成这些事：

- 重做或重构 README，让首页更像正式对外页面
- 检查安装、使用、示例、截图、License 等常见遗漏
- 在策略明确时顺带处理 `LICENSE`
- 在边界清晰时自动完成 Git 收尾并推送到远程仓库

## 对谁有用

它更适合这些人：

- 独立开发者：项目写完了，但仓库门面还很毛糙
- AI 编程用户：代码已经有了，需要 AI 帮你把门面和交付材料补齐
- 开源作者：想把仓库整理到能发链接给别人看的状态
- 做 skill、工具、小产品的人：需要一个能介绍、能安装、能上手的首页
- 做阶段性交付的人：每完成一个模块，就快速产出阶段性 README

## 适用范围

它适合处理这些对象：

- 单项目仓库
- 多项目仓库
- 技能集合仓库
- 仓库根 README
- 子项目 README
- 单个 skill README

它最擅长做到这一步：

- 最低：把 README 整理清楚
- 常规：把 README 和 LICENSE 一起补齐
- 完整：把门面材料整理完，并在条件明确时自动提交并推送

以下情况更适合人工主导：

- 产品定位本身还没想清楚
- 授权策略需要你自己判断
- 工作区里混着大量无关改动
- 你要的是深度品牌设计或强营销文案

## 使用效果

正常情况下，跑完这一轮后，你拿到的不是一段零散文案，而是一套更完整的门面结果：

- 一个更像正式项目首页的 README
- 更清楚的安装和最短调用方式
- 更完整的使用说明、示例入口和遗漏项检查
- 在策略明确时同步好的 License 说明
- 在边界清晰时已经完成的 Git 提交和远程推送

## 它是怎么工作的

你可以把它理解成一个门面送审流程：
AI 先整理材料、补齐缺项、送到关口；人不亲自搬运和制单，只负责审核、放行或打回。
Git 负责让文件安全，人负责把住质量，AI 负责让这套流程更有效率。

这个 skill 不是盲写，而是按下面的顺序工作：

1. 先调查当前仓库状态并推荐执行模式
2. 判断当前上下文是否充分
3. 如果上下文不足，先读取工作区里的关键文件
4. 自动判断仓库形态和本次生成层级
5. 基于上下文判断项目类型
6. 生成或重构 README
7. 评测 README，并决定是否放行
8. 通过后自动处理 `LICENSE`
9. 在 README 和 `LICENSE` 都通过后自动提交并推送到远程仓库

也就是说，它不是要求你必须先把上下文喂满，而是：

- 有上下文，就直接进入调查与模式推荐，再继续执行
- 上下文不够，就先补最小必要上下文，再进入调查与执行

## 它会先判断什么

默认优先由 AI 自己判断，而不是先把这些问题丢给你：

- 当前是单项目仓库、多项目仓库，还是技能集合仓库
- 这次要写的是仓库根 README、子项目 README，还是单个 skill README
- 当前项目更像库、CLI、网站、Skills、理论框架，还是文档教程

只有当判断结果会明显影响输出方向，且当前上下文无法稳定推断时，它才会最小化向你确认。

## 它会先推荐怎么执行

这个 skill 不应该一启用就直接改文件。更稳的方式是：

1. 先调查
2. 再推荐执行模式
3. 你确认一次
4. 后面自动推进

默认优先推荐三种模式之一：

- `审核后自动`：先给你一张建议卡，你确认一次后自动跑完
- `自动流`：策略已明确时，直接从 README 跑到 LICENSE 和 Git 收尾
- `手动流`：每一步都停下来确认

通常默认推荐的是 `审核后自动`，因为它既不会一上来就越权改仓库，也不会让你手动指挥每一步。

## 自动工作流

这个 skill 更适合“先调查并选模式，再自动跑完”，而不是 README 做完后再手动指挥每一步。

你只需要尽量先定这几件事：

- README 档位：快速 / 标准 / 发布
- LICENSE 策略：沿用现有 / 固定许可证 / 由 AI 推荐并直接应用 / 始终询问
- 推送策略：不推送 / 只提交本地 / 远程明确时自动推送

默认顺序是：

1. 先生成 README
2. 先做一次 README 评测
3. 若评测通过，再自动检查 LICENSE
4. 若评测需要小修，就先自动修一轮再复评
5. 若 README 和 LICENSE 都就绪且通过，就自动执行 Git 收尾

只有当许可证策略不明确，或远程目标不明确时，它才会停下来问你一次。

如果工作区里还混着其他不属于这次门面整理的改动，它也不应该直接自动提交；这时会先停下来让你确认边界。

## README 评测与放行

README 做出来之后，不直接继续往下跑，而是先过一次审。

评测重点是：

- 有没有先回答读者最关心的问题
- 有没有写对当前仓库层级和项目类型
- 安装、使用、示例是不是能直接用
- 有没有明显遗漏、冲突或过度承诺

评测结果分三种：

- `绿色`：通过，继续后续流程
- `黄色`：基本可用，但先自动修一轮再复评
- `红色`：方向不对或信息不足，先停下来给你确认

只有 README 评测通过后，它才继续处理 `LICENSE` 和 Git 收尾。

## 它会优先补哪些上下文

当上下文不足时，它会优先看这些关键文件，而不是无差别扫全仓库：

- 顶层 `README.md`
- `package.json`
- `pyproject.toml`
- `requirements.txt`
- `Cargo.toml`
- `go.mod`
- 主要源码目录
- 文档目录
- 示例目录
- 关键配置文件

## 它会检查哪些遗漏

README 生成完成后，它不会立刻停下，而会继续检查：

- 项目名称与一句话介绍是否完整
- 安装方式是否缺失
- 使用方法是否缺失
- 示例、截图、演示入口是否缺失
- 是否已有 `LICENSE`
- README 中的许可证说明是否缺失或冲突
- 当前目录是否为 Git 仓库
- 是否已配置远程仓库
- 当前分支是否已有明确的远程推送目标
- 是否存在可提交改动
- 当前工作区是否混有无关改动

## 案例

### 案例 1：技能集合仓库

`creator-publish-workflow` 本身也是一个典型例子。它不是单个 skill 仓库，而是一条由多个 skill 组成的工作流：

- 仓库根 README：作为整个工作流的入口页
- `项目门面生成器/README.md`：作为单个 skill 的介绍页
- `项目门面生成器/skill.md`：作为这个 skill 的执行规则页
- `秋知X推文发帖生成器/README.md`：作为下一步传播 skill 的介绍页

处理这种仓库时，这个 skill 会先把层级分清，再分别组织内容，而不是把整个仓库误写成单个项目。

### 案例 2：单项目仓库

如果一个仓库只服务一个项目，那么根 README 直接就是项目门面，重点放项目介绍、安装、使用、示例和 License。

### 案例 3：多项目仓库

如果一个仓库包含多个独立项目或模块，那么根 README 应该写成总入口和项目地图；每个子项目再分别生成自己的 README。

## 输出风格

默认支持两档：

- `极简`：少装饰，正文优先
- `标准`：适量徽章、清晰分节、适度表格与引用

如果项目适合对外展示，标准风格下可优先使用“封面式开头”：

```md
横幅图或品牌图
H1 标题
按钮式徽章行
1 到 2 段高密度介绍
```

## Skills 项目的特殊规则

如果目标本身是一个 skill，那么 README 默认要带 `安装` 部分。  
并且安装方式优先输出两种：

1. 命令安装
2. 让 AI 帮你安装

这两种方式都会尽量直接从上下文中推断出：

- 仓库地址
- skill 名称
- 安装命令

避免把本来 AI 能判断的东西继续丢给用户。

## 设计原则

```text
人做决策
AI 做分析与执行
优先利用已有上下文
上下文不足时自动补齐
README 不是终点，门面收尾才算完成
先调查并推荐模式，再自动推进
策略先定一次，后续自动推进
提交边界不清时先停下来确认
```
