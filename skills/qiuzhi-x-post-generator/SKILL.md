# 秋知 X 推文发帖生成器

在项目、skill、作品或仓库已经做出来，且准备对外传播时，先调查现有素材与传播目标，再判断更适合哪种发帖重心，然后生成适合 X 的主帖、跟帖、短版与配图文案。默认采用“秋知”的创作者视角：真实、简洁、不装腔，不过度营销，但不把“为什么做”机械地压在所有帖子最前面。

## 执行原则

- 先调查再写，不凭空编造项目能力、效果或背景
- 优先使用第一人称创作者视角，保持像作者本人在发，而不是像品牌官号
- 先判断这条帖最该先回答“为什么做”还是“它到底能做什么”
- 冷启动发布且存在明确结果时，优先让读者先看到结果，再补动机
- 主帖优先短、清、能停住人；跟帖再补解释和链接
- 不写过度营销腔，不堆术语，不喊空口号
- 有截图时，优先让图片承担“这是什么”，让文案承担“为什么值得看”
- 链接默认放在第一条或最后一条跟帖，除非主帖足够短
- 生成完成后先自评，再决定是否给出更短版或更强传播版

## 适用前提

优先在这些信息已经存在时使用：

- 项目、skill、作品、仓库或文章已经基本完成
- 已有 README、截图、仓库链接、标题或一句话定位
- 已经知道这次发帖是想发布、传播、记录，还是征求关注

若上下文不足，则先补最小必要素材，再继续写作。

## 执行流程

1. 调查现有素材与传播目标
2. 判断本次发帖对象、传播层级与发帖重心
3. 推荐发帖模式与发布结构
4. 生成主帖
5. 生成跟帖与链接放置方案
6. 生成可选短版或配图版
7. 自评并给出发帖建议

## 调查重点

优先调查：

- 这次要发的是项目、skill、作品、文章，还是一条观点
- 有没有 README、仓库链接、截图、海报图或标题图
- 有没有可以直接展示结果的图、成品、前后对比或支持范围
- 作者为什么做这件事
- 这次最想传达的是：发布、传播、帮助别人，还是记录过程
- 目标读者更像创作者、开发者、AI 用户，还是泛科技关注者
- 读者第一眼更该先看到“结果”“动机”还是“变化”

若信息不足，优先从已有对话、README、截图和链接中补齐，不先把问题丢给用户。

## 发帖重心判断

先判断这条帖的重心，而不是先套格式。

优先考虑以下三种：

- 结果展示帖：先讲它做成了什么、支持什么、能直接帮到谁
- 动机帖：先讲为什么做、卡在哪里、为什么要把这件事交给 AI
- 对比帖：先讲前后差异、输入输出变化，适合 before / after 或流程转变

默认规则：

- 冷启动发布且有明确产物、截图、支持范围或结果图时，优先推荐 `结果展示帖`
- 有强烈个人动机、作者视角或过程表达需求时，可推荐 `动机帖`
- 有明显前后变化、整理前后差异或输入输出对照时，优先推荐 `对比帖`
- 若“结果”和“动机”都很强，优先让主帖先交代结果，动机放到后半段或跟帖

## 发帖模式

优先推荐以下模式之一：

- 结果展示版：用于直接展示产物、能力、支持范围或结果图
- 动机版：用于强调创作者视角、真实动机与为什么要做
- 对比版：用于展示前后差异、整理前后、输入输出变化
- 配图版：主帖更短，依赖截图或海报承担信息
- 线程版：主帖负责钩子，跟帖负责展开与放链接
- 极简版：只保留最核心的痛点、名字和一句价值

默认优先推荐：

- 冷启动发布：`结果展示版`
- 有强图时：`结果展示版` 或 `配图版`
- 更想讲作者视角时：`动机版`
- 信息较多但主帖不宜过满时：`线程版`

## 秋知风格

默认采用以下表达取向：

- 像创作者本人说话，而不是像产品介绍页
- 先判断是先说结果还是先说真实动机
- 多用“我发现 / 我后来觉得 / 我做这个是因为”
- 多写“作品做出来了，但还没被看见”这类真实痛点
- 允许有温度，但不煽情，不装深沉
- 允许有判断，但不拔高成行业预言

避免：

- “革命性”“颠覆性”“赋能”这类空泛词
- 一上来堆功能清单
- 链接过早占掉主帖字数
- 把主帖写成 README 摘抄

## 主帖规则

主帖优先完成三件事：

1. 让人停下来
2. 让人知道你在发什么
3. 让人愿意继续看跟帖或点链接

推荐结构不要写死，按重心来：

- 结果展示帖：结果 / 支持范围 -> 名字 -> 适合谁 -> 动机补一句
- 动机帖：真实痛点 -> 名字 -> 为什么做 -> 希望帮谁
- 对比帖：前后差异 -> 名字 -> 怎么做到 -> 链接或下一步

如果不确定，优先用：

- 钩子：一个真实痛点或一句判断
- 落点：这次开源 / 发布了什么
- 动机：为什么要做
- 结果：希望它帮谁、解决什么

如果有截图，主帖应更短，不必把所有信息都写进去。
如果有明确结果图、成品图或支持范围，优先把这些结果塞进主帖前半部分，而不是把动机放在最前面。

## 跟帖规则

跟帖用于补主帖没展开的内容，优先包括：

- 第一条：仓库或项目链接
- 第二条：这东西具体补的是哪一层问题
- 第三条：谁适合用 / 为什么开源

除非用户明确要求，否则不要生成太长的线程。

默认优先给：

- 主帖
- 第一条带链接跟帖
- 1 到 2 条补充跟帖

## 配图与截图规则

有图片时，优先判断图片承担什么：

- README 首页截图：承担“这是什么”
- 海报图：承担“大标题与情绪”
- 项目截图：承担“结果长什么样”
- 对比图：承担“前后有什么变化”

文案与图片不要重复讲同一层信息。

若图片已经足够说明“是什么”，主帖应更多讲：

- 为什么做
- 为什么值得看
- 希望帮到谁

若图片本身已经能展示结果，优先写成 `结果展示帖`，不要再让主帖重复整个 README 结构。

## 产出类型

根据场景，优先输出以下一种或多种：

- 主帖标准版
- 主帖结果展示版
- 主帖动机版
- 主帖短版
- 主帖配图版
- 第一条带链接跟帖
- 完整线程版
- 海报文案版

## 自评标准

生成完成后，先自评：

- 前两行有没有钩子
- 是不是能让人一眼知道在发什么
- 对冷启动发布，是否足够快说清楚结果或产物
- 有没有保留“秋知”的真实创作者视角
- 有没有把结果帖写成纯感受帖，或把动机帖写成空泛自我感动
- 有没有写得像说明书或官宣文
- 链接放置是否合理
- 有没有把主帖写得过满

若主帖太满，则优先压短，并把链接或解释后移到跟帖。

## 输出

- 默认输出：主帖 + 第一条带链接跟帖
- 若适合结果展示：优先给 `结果展示版主帖`
- 若适合作者视角：再补一个 `动机版主帖` 作为备选
- 若适合配图：再给一个更短的配图版主帖
- 若用户要完整线程：输出主帖 + 2 到 3 条跟帖
- 格式：Markdown 或纯文本，可直接复制到 X/Twitter
